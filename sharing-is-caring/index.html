Now That We're on the Same Page

Lets start sharing code so we can stop repeating ourselves.

Models seems like a great place to start, right? We're going to need two things in order to make this work:

  1. A new Backbone.Sync method we can use on the server end
  2. A method of knowing which environment we're in

So we've got a usable back end javascript environment. And we've got tools that level the playing field on the client (es5shim, jquery, modernizr, stitch, browserify, require.js - the list goes on), so lets start sharing more by starting with our data models. In order to do this we're going to need to solve for two immediate issues:
  1. Make a backbone.sync to write to disc since it defaults to AJAX
  2. Know when to inject that version into our model

Backbone.Sync is Designed for this

Backbone.Sync has a very easy signature to replace: 


Backbone.Sync = function(method, model, options);

So we need to take that signature and just write it so it reads/writes from the datastore. For a DB interface, we'll use MongoJS

And since it returns a jQuery $.Deferred on the front end, we'll make this "less surprises" and give you back a `then`-able promise from Q.

One of the best decisions was to separate sync from the methods which rely on it: save and fetch. This allows us to easily replace it whenever we need to provide alternate functionality so making backbone sync to socket.io, localStorage, Redis, whatever, becomes a simple method swap.
And following the rule of less surprises (Check your handbook, page 42) we are going to mimic default sync: return a then-able and resolve that then-able with pure JSON

var callback = function(err, results){
	if(err){
		d.reject(err);
	} else {

			d.resolve(model.toJSON());
		} else {
			d.resolve(results[0]);

}

Ahh. You're all looking at that 1 in there eh? well...MongoDB WTF? If you perform an upsert (updating data in place you've already stored), you get back 1. Seriously. Try it. So if we upsert, we'll just return the model data since it's already got all the info we need, we just wanted to verify it's now written to mongo.

And see that returning the JSON representation of the model when we resolve? This what the standard sync does, so, again, less surprise == better.

We're Almost Home

Now onto number two: determining environment. There are a couple of things we need to consider for this:

  • Is process available?
  • Is exports available?
  • If we expect Browserify to work, we need to check process.browser as well

OK. Phew. Still with me? That was the complex part of the application, so we're in the home stretch. BUT! This next part is almost the most important part -- we need to explain to our code when to attach our new sync method or when to use the default (or even a third implementation!) We check for process, exports and process.browser.

Attach the Sync on the Server


if(typeof process !== 'undefined' &&
   !process.browser &&
   typeof exports !== 'undefined'){
    var sync = require('new_sync');
    Backbone.sync = sync(server, collection)
}

Don't forget: we need to make sure the sync method gets the datastore configuration!

A bit of gnarly if statement up there, but this should take care of all our scenarios -- we need to check if process exists, and then for browserify, we check for "browser" and finally lets see if exports exists for shits and giggles.

And Use that to Return a Custom Model


module.exports = function(server, collection){
	if(typeof process !== 'undefined' &&
   	   !process.browser &&
   	   typeof exports !== 'undefined'){
    	var sync = require('new_sync');
    	Backbone.sync = sync(server, collection);
	}

	return Backbone.Model.extend({idAttribute: '_id'});
}

We will return a custom model back, setting idAttribute correctly for mongodb, just using the basic Backbone.Model.extend method.
